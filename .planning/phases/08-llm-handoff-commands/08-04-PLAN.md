/**
 * Trigger :yank wireframe for the active screen (D-209 side-effect pattern).
 * Writes the ASCII content-variant wireframe to the OS clipboard.
 * Key: Ctrl+Y (\x19 is Ctrl+Y in many terminals — verify against existing Ctrl+Y=redo binding)
 *
 * NOTE: Ctrl+Y is currently bound to redo (\x19). Use Ctrl+W (\x17) for yank instead,
 * or check the existing handleInput block and pick a key that is NOT already bound.
 * Safe choices (not bound in the current handleInput block):
 *   Ctrl+W = \x17, Ctrl+K = \x0b, Ctrl+J = \x0a (but \x0a is Enter in some contexts)
 * Decision: use Ctrl+W (\x17) for yank-wireframe, following the "W = wireframe" mnemonic.
 */
private triggerYankWireframe(): void {
  const state = this.store.getState();
  const screenId = this.activeScreenId ?? "";
  if (!screenId) {
    if (this.emitStatusTimer !== null) clearTimeout(this.emitStatusTimer);
    this.emitStatus = { message: "No screen selected", ok: false };
    this.tui?.requestRender();
    this.emitStatusTimer = setTimeout(() => {
      this.emitStatus = null;
      this.emitStatusTimer = null;
      this.tui?.requestRender();
    }, 3000);
    return;
  }
  void runYankWireframe(state.spec, screenId).then((result) => {
    if (this.emitStatusTimer !== null) clearTimeout(this.emitStatusTimer);
    this.emitStatus = { message: result.message, ok: result.ok };
    this.tui?.requestRender();
    this.emitStatusTimer = setTimeout(() => {
      this.emitStatus = null;
      this.emitStatusTimer = null;
      this.tui?.requestRender();
    }, 3000);
  });
}

/**

Resolve RESEARCH Open Question A2: How do side-effect commands surface in the palette?

**Analysis first:** Read src/canvas/palette/index.ts fully. Identify:
- Does it have any mechanism other than COMMANDS enumeration?
- Where does `store.apply(commandName, args)` get called after arg collection?
- Is there an `onClose` or `onSubmit` callback that could route to a side-effect handler?

**Implementation decision:** The palette currently only dispatches to `store.apply()` after arg collection. D-208 says the three handoff commands should be palette-discoverable. D-209 says they follow the emit-maestro shape (not Command<T>).

There are two viable approaches:

**Approach A — Extend CommandPalette to accept a sideEffectCallback:**
Add an optional `onSideEffect?: (name: string, args: Record<string, string>) => void` parameter to the `CommandPalette` constructor. When a command is listed in a `SIDE_EFFECT_COMMANDS` constant (not COMMANDS), the palette calls `onSideEffect(name, args)` instead of `store.apply()`.

This requires changes to BOTH `src/canvas/palette/index.ts` AND `src/canvas/root.ts`.

**Approach B — Hardcode in handleInput only (no palette discoverability):**
The three handoff commands are only reachable via direct key bindings (Ctrl+W for yank; prompt and extract have no direct shortcut). The palette does NOT show them. This violates D-208.

**Required approach:** Approach A (D-208 is locked — palette-discoverable is non-negotiable).

**Concrete implementation for Approach A:**

In src/canvas/palette/index.ts:
- Add a `SIDE_EFFECT_ENTRIES` type: `{ name: string; description: string; args: Array<{ name: string; prompt: string; options?: string[] }> }[]`
- Add an optional `sideEffects?: SideEffectEntry[]` parameter to `CommandPalette` constructor
- In the filter list, show side-effect entries mixed with COMMANDS entries
- When a side-effect entry is selected and all args are collected, call a provided `onSideEffect(name, args)` callback instead of `store.apply()`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Palette arg-prompt → trigger methods | screenId and target strings arrive from user keyboard input via palette arg collection |
| Trigger methods → runner functions | screenId passed to runYankWireframe / runExtractScreen without additional validation in root.ts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-01 | Tampering | dispatchSideEffect → runExtractScreen path | mitigate | Path sanitization already in runExtractScreen (extract-screen.ts): `/[^a-z0-9_-]/g` + `basename()`. root.ts does not need to re-sanitize — defense-in-depth happens at the IO boundary. |
| T-8-07 | Spoofing | Palette side-effect command injection | accept | HANDOFF_COMMANDS entries are hardcoded constants in root.ts — not user-provided. The only user-provided input is the arg values (screenId, target), which are sanitized at the runner layer. |
| T-8-08 | Denial of Service | Ctrl+W rapid-fire invocations | accept | emitStatusTimer is cleared before each new trigger — rapid invocations clear the previous timer and reset the 3s countdown. No accumulating timers, no runaway state. |
</threat_model>
