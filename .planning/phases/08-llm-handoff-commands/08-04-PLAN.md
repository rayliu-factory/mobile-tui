**Why COMMANDS (not a parallel table):** The palette (src/canvas/palette/index.ts) enumerates `Object.values(COMMANDS)` exclusively — this is the only palette surface. D-208 requires palette discoverability. Therefore these entries MUST be in COMMANDS.

**The Command<T> shape for side-effect commands:**
- `apply()` returns the UNCHANGED spec (no spec mutation — these are read-only side effects)
- `apply()` fires the runner via `void runXxx(...).then(callback)` — fire-and-forget
- `invert()` returns the UNCHANGED spec — no-op (side effects cannot be undone)
- An undo entry IS pushed to the stack by the store (this is unavoidable), but invoking undo calls `invert()` which changes nothing visible — effectively transparent

**How runner results reach the status line:** Each Command entry needs a reference to `RootCanvas.notifySideEffectResult` to surface results via the emitStatus pattern. Pass this as a module-level mutable slot on each command object, set by RootCanvas after store creation.

**Step 1: Add Command<T> entry to src/editor/commands/yank-wireframe.ts**

Add at the bottom of the file (after the existing `runYankWireframe` export):

```typescript
import { z } from "zod";
import type { Command } from "../types.ts";

const yankWireframeArgsSchema = z.object({
  screenId: z.string(),
});

export const yankWireframeCommand: Command<typeof yankWireframeArgsSchema> & {
  _onResult?: (r: { ok: boolean; message: string }) => void;
} = {
  name: "yank-wireframe",
  argsSchema: yankWireframeArgsSchema,
  apply(spec, _astHandle, args) {
    // Fire runner as side effect — result delivered via _onResult callback (D-211)
    void runYankWireframe(spec, args.screenId).then((result) => {
      yankWireframeCommand._onResult?.(result);
    });
    return { spec, inverseArgs: null }; // unchanged spec — no mutation
  },
  invert(spec, _astHandle, _inverseArgs) {
    return { spec }; // no-op — side effects cannot be undone
  },
  _onResult: undefined,
};
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Palette arg-prompt → store.apply → command.apply | screenId and target strings arrive from user keyboard input via palette arg collection, validated by argsSchema.safeParse before reaching apply() |
| command.apply → runner functions | screenId passed to runYankWireframe / runExtractScreen; path sanitization happens inside the runners |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-01 | Tampering | extractScreenCommand.apply → runExtractScreen path | mitigate | Path sanitization already in runExtractScreen (extract-screen.ts): `/[^a-z0-9_-]/g` + `basename()`. command.apply does not re-sanitize — defense-in-depth happens at the IO boundary. |
| T-8-07 | Spoofing | Palette command injection via COMMANDS names | accept | COMMANDS entries are module-level constants. The only user-provided input is arg values (screenId, target), which are sanitized at the runner layer and validated by argsSchema.safeParse before reaching apply(). |
| T-8-08 | Denial of Service | Ctrl+W rapid-fire invocations | accept | emitStatusTimer is cleared before each new trigger — rapid invocations clear the previous timer and reset the 3s countdown. No accumulating timers, no runaway state. |
| T-8-09 | Tampering | _specFilePath module-level mutable slot | accept | Set once in RootCanvas constructor from `store.getState().filePath` — a value already loaded from the filesystem at startup. Not user-provided at runtime. |
</threat_model>
