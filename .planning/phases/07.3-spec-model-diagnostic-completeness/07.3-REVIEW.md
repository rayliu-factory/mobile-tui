# Phase 07.3: Code Review Report

**Reviewed:** 2026-04-20T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

These files implement the two-stage spec validator (Zod structural parse + cross-reference
pass), the migration chain runner, the parse entry point, and supporting fixtures and tests.
The overall architecture is solid: no-throw guarantees, clean diagnostic shapes, correct
RFC 6901 encoding, and thorough test coverage for cross-reference codes.

Three issues need attention before phase closes:

1. An infinite loop is possible in `runMigrations` if the MIGRATIONS array ever contains
   a cycle — there is no iteration cap.
2. `submit.entity` unresolved references are emitted with the wrong diagnostic code
   (`SPEC_UNRESOLVED_ACTION` instead of a more specific entity code), which is a semantic
   mismatch that confuses callers and the snapshot.
3. The migration target version in `parseSpecFile` is hardcoded to `"1"` rather than
   tied to `SCHEMA_VERSION`, which will silently break the migration gate when a v2 schema
   ships.

## Warnings

### WR-01: Infinite loop in `runMigrations` — no iteration cap

**File:** `src/migrations/index.ts:29`
**Issue:** The `while (v !== toVersion)` loop has no iteration limit. If the `MIGRATIONS`
array ever gains a cycle (e.g., `{from:"1",to:"2"}` + `{from:"2",to:"1"}`) and the caller
asks for a version reachable only via that cycle, the loop never terminates. This is a
latent DoS risk as the migrations table grows. The current single-entry table cannot trigger
it, but there is no guard to enforce that invariant as new migrations are added.

**Fix:**
```typescript
// Add an iteration cap equal to MIGRATIONS.length + 1 before the while loop.
const maxSteps = MIGRATIONS.length + 1;
let steps = 0;
while (v !== toVersion) {
  if (++steps > maxSteps) {
    return {
      result: spec,
      diagnostic: {
        code: "SPEC_UNSUPPORTED_VERSION",
        severity: "error",
        path: "",
        message: `Migration cycle detected between v${fromVersion} and v${toVersion}`,
      },
    };
  }
  // ... rest of loop body unchanged
}
```

### WR-02: Wrong diagnostic code for `submit.entity` unresolved reference

**File:** `src/model/cross-reference.ts:362`
**Issue:** When a `submit` action references an entity that is not in `data.entities`, the
emitted diagnostic uses code `SPEC_UNRESOLVED_ACTION`. This is semantically wrong — the
unresolved reference is an entity, not an action. Consumers triaging diagnostics by code
will be misled: they will see `SPEC_UNRESOLVED_ACTION` pointing at an `.../entity` path
and interpret it as an action-id problem.

The `navigate.screen` case (line 350) correctly uses `SPEC_UNRESOLVED_SCREEN`. The same
domain-specific code discipline should apply here. A new code `SPEC_UNRESOLVED_ENTITY`
would be the clean fix; alternatively reuse `SPEC_UNRESOLVED_ACTION` only for action-id
references. The test at `cross-reference.test.ts:123` currently asserts the wrong code
and would need updating.

**Fix:**
```typescript
// cross-reference.ts — submit case, line 361-366:
case "submit":
  if (!declaredEntities.has(action.entity)) {
    diagnostics.push({
      code: "SPEC_UNRESOLVED_ENTITY",   // was SPEC_UNRESOLVED_ACTION
      severity: "error",
      path: pathToJsonPointer(["actions", actionId, "entity"]),
      message: `submit.entity "${action.entity}" not declared in data.entities`,
    });
  }
  break;
```

Update the matching assertion in `cross-reference.test.ts:124`:
```typescript
expect(
  diags.some((d) => d.code === "SPEC_UNRESOLVED_ENTITY" && d.path.includes("/entity")),
).toBe(true);
```

Also update the snapshot in `tests/__snapshots__/malformed.test.ts.snap` and
`CROSSREF_CODES` in `tests/malformed.test.ts` if `SPEC_UNRESOLVED_ENTITY` is added as a
covered code.

### WR-03: Migration `toVersion` hardcoded to `"1"` in `parseSpecFile`

**File:** `src/serialize/parse.ts:169`
**Issue:** `toVersion` is hardcoded as the string literal `"1"` rather than derived from
`SCHEMA_VERSION`. When a v2 schema is introduced (which `src/migrations/v1_to_v2.ts`
explicitly anticipates), files with `schema: mobile-tui/1` will be migrated toward `"1"`,
which is a no-op, while the validator expects v2 shape. The symptom will be subtle:
`parseSpecFile` will silently produce a structurally-invalid spec after migration instead
of running the `1→2` migrator. The migration code that exists will never fire in the
production path.

**Fix:**
```typescript
// parse.ts — step 8.5, import SCHEMA_VERSION and use it as the target:
import { SCHEMA_VERSION } from "../model/version.ts";

// ...
const toVersion = SCHEMA_VERSION.replace("mobile-tui/", "") as SpecVersion;
if (fromVersion !== toVersion) {
  const { result: migrated, diagnostic: migrationDiag } = runMigrations(
    partition.knownSubset,
    fromVersion,
    toVersion,
  );
  // ...
}
```

### IN-01: `present.overlay` missing-screen case uses `SPEC_UNRESOLVED_ACTION` code

**File:** `src/model/cross-reference.ts:384`
**Issue:** When `present.overlay` names a screen that does not exist in `screens`, the
diagnostic code is `SPEC_UNRESOLVED_ACTION` (line 384). The parallel case for
`navigate.screen` uses the more precise `SPEC_UNRESOLVED_SCREEN`. A missing overlay is a
screen reference, not an action reference. The mismatch is minor (both resolve to
error-severity) but produces misleading paths for diagnostic consumers.

**Fix:**
```typescript
// cross-reference.ts — present case, line 382-387:
if (!overlay) {
  diagnostics.push({
    code: "SPEC_UNRESOLVED_SCREEN",   // was SPEC_UNRESOLVED_ACTION
    severity: "error",
    path: pathToJsonPointer(["actions", actionId, "overlay"]),
    message: `present.overlay "${action.overlay}" not declared in screens`,
  });
}
```

### IN-02: Redundant optional chaining in `test_flows` loop

**File:** `src/model/cross-reference.ts:447-449`
**Issue:** The loop guard is `(spec.test_flows ?? []).length` — if `spec.test_flows` is
null/undefined, the array is `[]` and the loop body never executes. Yet the inner
dereference on line 449 uses `spec.test_flows?.[fi]`, which repeats the optional access
unnecessarily. After the `?? []` guard, `spec.test_flows` being absent means the loop
body is unreachable; the inner optional chain adds noise and hides that reasoning.

**Fix:**
```typescript
const testFlows = spec.test_flows ?? [];
for (let fi = 0; fi < testFlows.length; fi++) {
  const flow = testFlows[fi];
  if (!flow) continue;
  // ...
}
```

### IN-03: Migration test bypasses public API

**File:** `src/migrations/index.test.ts:2`
**Issue:** The test imports `validateSpec` directly from `"../model/invariants.ts"` rather
than from the public barrel `"../src/index.ts"` (or the model's own barrel). Bypassing
the public API means the test does not verify that `validateSpec` is correctly re-exported,
and a refactor that moves the function without updating the barrel would leave this test
green while consumers break.

**Fix:** Import from the model barrel or the package public entry:
```typescript
// Replace:
import { validateSpec } from "../model/invariants.ts";
// With:
import { validateSpec } from "../index.ts";
// or
import { validateSpec } from "../model/index.ts";
```
