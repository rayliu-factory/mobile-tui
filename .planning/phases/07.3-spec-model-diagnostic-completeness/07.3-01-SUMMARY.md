## What Was Built

### WR-01: SPEC_DUPLICATE_SCREEN_ID
Replaced `new Set(spec.screens.map(s => s.id))` one-liner with a `Map<string, number>` scan that emits `SPEC_DUPLICATE_SCREEN_ID` at the second-occurrence path (`/screens/N/id`) with a message containing the duplicate id and the first-seen path.

### WR-02: SPEC_DUPLICATE_ENTITY_NAME
Same pattern for entities: `Map<string, number>` scan emitting `SPEC_DUPLICATE_ENTITY_NAME` at `/data/entities/N/name`.

### WR-03: RFC 6901 decode in resolveJsonPointerPrefix
Added `.map(decodeSegment)` after `.split("/")` in `resolveJsonPointerPrefix` so that pointer segments like `A~1B` correctly decode to `A/B` before entity lookup. Exported the function for direct unit testing.

### IN-04: SPEC_UNRESOLVED_SCREEN semantic rename
Renamed `SPEC_UNRESOLVED_ACTION` to `SPEC_UNRESOLVED_SCREEN` at four locations: `navigate.screen`, `navigation.root`, `nav edge.from`, `nav edge.to`. The `nav edge.trigger` check correctly retains `SPEC_UNRESOLVED_ACTION` (it checks the actions registry).

### IN-01: invalid_key ZOD_CODE_MAP entry
Added `invalid_key: "SPEC_INVALID_KEY"` to `ZOD_CODE_MAP` in `zod-issue-adapter.ts`.

### IN-02: JSON.stringify(undefined) comment
Added clarifying comment above the `typeof serialized === "string"` guard in `invariants.ts` explaining why the guard is intentional.

### IN-03: resolveJsonPointerPrefix docstring update
Removed the "no decode applied" caveat and replaced with documentation that `decodeSegment` is applied.

### IN-06: Object.entries ordering comment
Added comment above `Object.entries(spec.actions)` explaining ES2015+ insertion-order determinism.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] WR-03 test approach incompatible with Zod PascalCase constraint**
- **Found during:** Task 1, Step B (GREEN)
- **Issue:** The plan's WR-03 tests used entity names `"A/B"` and `"A~B"` with `SpecSchema.parse()`, but Zod's entity name regex `/^[A-Z][A-Za-z0-9]*$/` (PascalCase, letters/digits only) rejects names containing `/` or `~`. The `parse()` call threw `ZodError` before `crossReferencePass` was ever reached.
- **Fix:** Exported `resolveJsonPointerPrefix` and rewrote the WR-03 tests to call it directly with a synthetic Spec-shaped object. This correctly tests the RFC 6901 decode behavior without fighting the schema constraint. Added a third test verifying the baseline (normal pointer resolution still works).
- **Files modified:** `src/model/cross-reference.ts` (export added), `src/model/cross-reference.test.ts` (test rewrite)
- **Commit:** b70af86

**2. [Rule 1 - Bug] Existing navigate.screen test still asserted SPEC_UNRESOLVED_ACTION**
- **Found during:** Task 1, Step C (GREEN verification)
- **Issue:** The existing test `"navigate.screen pointing to unknown screen"` in the `SPEC_UNRESOLVED_ACTION` describe block still asserted `d.code === "SPEC_UNRESOLVED_ACTION"` for the `/screen` path. After IN-04 renamed that emission to `SPEC_UNRESOLVED_SCREEN`, the test failed.
- **Fix:** Updated the assertion to `SPEC_UNRESOLVED_SCREEN` to match the new semantic code.
- **Files modified:** `src/model/cross-reference.test.ts`
- **Commit:** b70af86
