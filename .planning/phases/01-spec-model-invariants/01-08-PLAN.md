<implementation_strategy>
**Fixture form note (Phase 1 vs Phase 2 scope):** Fixtures in Phase 1 use the **triple form** — `label`, `action`, and `testID` authored as separate YAML keys on each interactable component — per CONTEXT.md D-04. The sigil syntax `[Label →action test:id]` is **Phase 2's concern**; Phase 2 ships the parser that normalises sigil → triple on read and the serializer that re-emits sigil on write. Phase 1 stores and validates the structured triple directly. This ensures Phase 1 can ship a green `validateSpec()` without a sigil parser. A future reviewer comparing fixtures to the Phase 2 canonical wire format should expect the sigil syntax only after Phase 2 lands; it is NOT a fixture bug that Phase 1 fixtures write separate YAML keys.

Every fixture's Markdown body MUST include a one-line header comment to make this explicit to downstream readers:
```
<!-- Phase 1 fixture: triple-form YAML; sigil parser lands in Phase 2 -->
```

    function collectTestIDsAndScreenIds(spec: any): Set<string> {
      const ids = new Set<string>();
      for (const screen of spec.screens) {
        ids.add(screen.id);
        function walk(node: unknown): void {
          if (!node || typeof node !== "object") return;
          const n = node as Record<string, unknown>;
          if (typeof n.testID === "string") ids.add(n.testID);
          for (const v of Object.values(n)) {
            if (Array.isArray(v)) v.forEach(walk);
            else if (v && typeof v === "object") walk(v);
          }
        }
        for (const key of ["content", "empty", "loading", "error"] as const) {

          if (variant && Array.isArray(variant.tree)) variant.tree.forEach(walk);

      }
