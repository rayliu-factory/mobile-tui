# Phase 6.1 Plan 06.1-02: Fix DataStep entity persistence on wizard advance

**One-liner:** Wires store.apply("add-entity") calls into FormPane.tryAdvance() at stepIndex 5 so entity names collected by DataStep land in spec.data.entities before wizard graduation.

## What Was Built

Closed Gap 2 (WIZARD-02 / WIZARD-03): the DataStep in `form-pane.ts` collected entity names in internal state but never called `store.apply("add-entity", ...)`. Entities were silently discarded on wizard advance. The fix loops over `dataStep.getItems()` in the `stepIndex === 5` branch of `tryAdvance()`, calling `store.apply("add-entity", { name, fields: [{ name: "id", type: "string" }] })` for each name. A failing apply sets `this.error` and blocks advance.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] vi.waitFor compatibility with vitest 4.x**
- **Found during:** GREEN verification
- **Issue:** `vi.waitFor(() => condition)` resolved immediately on first check without polling, causing Test 1 to fail with 6ms duration (onAdvance called async after 100ms of mock store work). Tests 2 and 4 had the same pattern.
- **Fix:** Replaced `vi.waitFor` with `await new Promise((resolve) => setTimeout(resolve, 100))` to give the async `tryAdvance` chain time to complete.
- **Files modified:** src/wizard/panes/form-pane.test.ts
- **Commit:** dd49c8c (included in GREEN commit)

**2. [Rule 2 - Missing] EntityName branded type in test spec construction**
- **Found during:** `npx tsc --noEmit` after GREEN implementation
- **Issue:** `spec.data.entities = entityNames.map((name) => ({ name, ... }))` — `name: string` not assignable to `EntityName` branded type.
- **Fix:** Added `name: name as EntityName` cast and `import type { EntityName }` import.
- **Files modified:** src/wizard/panes/form-pane.test.ts
- **Commit:** dd49c8c (included in GREEN commit)
