# Phase 6.1: Code Review Report

**Reviewed:** 2026-04-19T00:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Reviewed the parse pipeline, both CLI entry scripts, the FormPane orchestrator, and their test suites. The code is generally well-structured and follows the established threat-model conventions. No critical security vulnerabilities were found.

Four warnings were identified: a fire-and-forget async method that swallows errors silently in FormPane, an unchecked non-null assertion in canvas.ts, a resource-leak risk in wizard.ts (autosave not disposed on parse-failure early exit path), and a subtle DataStep input-pending data loss bug (typed-but-not-yet-entered text is silently dropped on Tab). Three info items cover test reliability, unused import re-export, and a missing-error edge case in the migration catch block.

## Warnings

### WR-01: `tryAdvance()` is fire-and-forget — errors after the first `await` are silently swallowed

**File:** `src/wizard/panes/form-pane.ts:99`

**Issue:** `handleInput` calls `void this.tryAdvance()`. Because `tryAdvance` is `async`, any unhandled rejection after the first `await` is silently dropped — it never reaches the `this.error` field, the render loop never shows the user feedback, and no diagnostic is surfaced. The pattern is intentional (cannot `await` inside a synchronous event handler), but there is no `.catch()` fallback to set `this.error`, meaning the method's own `try`-less branches can silently throw if `store.apply` rejects unexpectedly (e.g., the command registry is not initialised).

**Fix:**
```typescript
// form-pane.ts line 99
if (data === "\t") {
  void this.tryAdvance().catch((err: unknown) => {
    this.error = err instanceof Error ? err.message : "Unexpected error.";
  });
  return;
}
```

### WR-02: Non-null assertion on `parseResult.astHandle` is unchecked in `canvas.ts`

**File:** `scripts/canvas.ts:48`

**Issue:** `createStore` is called with `astHandle: parseResult.astHandle!`. The guard on line 35 (`if (!parseResult.spec)`) does not assert `astHandle` is non-null separately. Per the `ParseResult` contract in `parse.ts` (line 208): `astHandle` is `null` when `spec` is `null`, and non-null only when `spec` is non-null. The logical equivalence holds today, but the assertion hides the coupling. If a future change to `parseSpecFile` returns a non-null `spec` alongside a null `astHandle` (e.g., a partial-migration path), `createStore` receives `null` where it expects `AstHandle` and crashes without a useful error message.

**Fix:** Assert both at the guard boundary, or destructure with an explicit null check:
```typescript
if (!parseResult.spec || !parseResult.astHandle) {
  for (const d of parseResult.diagnostics) {
    process.stderr.write(`${d.severity} ${d.path}: ${d.message}\n`);
  }
  process.exit(1);
}
// Now both are guaranteed non-null — no assertion needed below
const store = createStore({
  spec: parseResult.spec,
  astHandle: parseResult.astHandle,
  filePath: specPath,
});
```

The same pattern applies to `scripts/wizard.ts:65` (`parseResult.astHandle!`).

### WR-03: `autosave` is created before the render guard in `wizard.ts` — not disposed on empty-render early exit

**File:** `scripts/wizard.ts:67`

**Issue:** `createAutosave(store, resolvedPath)` is called on line 67, which registers a `process.on("beforeExit", ...)` handler and subscribes to the store. If the subsequent empty-render check on line 93 fires (`lines.length === 0`), the function returns early without calling `autosave.dispose()`. The `beforeExit` listener is then never removed, constituting the T-04-06 memory-leak pattern that the comments themselves warn about.

**Fix:** Either call `autosave.dispose()` before the early return, or construct autosave after the render guard:
```typescript
// Option A — dispose on early exit
if (lines.length === 0) {
  process.stderr.write("error: wizard render produced empty output\n");
  autosave.dispose();
  process.exitCode = 1;
  return;
}
```

Note: `canvas.ts` does NOT have this issue — it constructs `autosave` (line 49) before the render guard (line 57) but also returns early without disposing. The same fix applies there.

### WR-04: DataStep silently drops pending (typed-but-not-submitted) input on Tab

**File:** `src/wizard/panes/form-pane.ts:147–165` / `src/wizard/steps/data.ts:38–41`

**Issue:** When the user types a partial entity name but does not press Enter before pressing Tab, `DataStep.handleInput("\t")` returns `{ kind: "advance", args: this.items }` where `this.items` does NOT include `this.inputValue`. The pending text is silently discarded. `FormPane.tryAdvance` at step 5 then calls `this.dataStep.getItems()` which likewise only returns the committed items. This is data loss from the user's perspective — they typed "Order", forgot to press Enter, hit Tab, and "Order" vanishes with no error shown.

`ScreensStep` has the same behaviour but it is arguably more harmful there because the user must have at least one screen (D-94) — but the guard only checks `this.items.length`, not `this.inputValue.length > 0`, so a user who typed a screen name without pressing Enter will see "At least one screen is required" with their text still visible, causing confusion.

**Fix for DataStep:** Auto-commit pending input on Tab before returning advance:
```typescript
// data.ts handleInput — Tab case
if (data === "\t") {
  const trimmed = this.inputValue.trim();
  if (trimmed.length > 0) {
    this.items.push(trimmed);
    this.inputValue = "";
  }
  return { kind: "advance", args: this.items };
}
```

Apply the same fix to `ScreensStep.handleInput` for the Tab case.

## Info

### IN-01: `setTimeout`-based async settlement in tests is fragile

**File:** `src/wizard/panes/form-pane.test.ts:75,97,117`

**Issue:** Tests use `await new Promise((resolve) => setTimeout(resolve, 50))` or `100ms` waits to let the fire-and-forget `tryAdvance()` promise settle. This is timing-dependent and will produce flaky failures on heavily loaded CI runners. The root cause is that `handleInput` is synchronous but calls an `async` method as fire-and-forget (WR-01 above).

**Fix:** The cleanest solution is to expose a `_lastAdvancePromise` field on `FormPane` for test purposes, or to refactor `tryAdvance` to return a `Promise` that callers can await when needed. Alternatively, use `await vi.runAllTimersAsync()` with Vitest's fake timers if the implementation is changed to use `queueMicrotask` internally.

### IN-02: `YAML` default import is re-exported from `parse.ts` solely to prevent tree-shaking removal — this is fragile

**File:** `src/serialize/parse.ts:243`

**Issue:** The comment explains that `YAML` is re-exported so "downstream callers import YAML via this module." Re-exporting a side-effect-free value to satisfy tree-shaker heuristics is fragile: any bundler upgrade or biome rule change could remove it. More importantly, callers who want `YAML` should import it directly from the `yaml` package; re-exporting it through `parse.ts` creates an unexpected coupling between the serialize module and the YAML package surface.

**Fix:** Remove the re-export and have each call site import `YAML` directly from `"yaml"`. If the intent is to ensure the import survives in the bundle, add a `// @ts-ignore tree-shaker` comment inline rather than re-exporting through a public interface.

### IN-03: Migration catch block discards the original error — diagnostic message uses a hardcoded format that may mislead

**File:** `src/serialize/parse.ts:173–185`

**Issue:** The `catch` block at line 173 discards the thrown error entirely (`catch { ... }`). The diagnostic message is always `No migration path from schema version "${fromVersion}" to "1"`, but `runMigrations` might throw for other reasons (e.g., an internal invariant violation in a migration function). The hardcoded message would then be misleading.

**Fix:** Capture the error and incorporate its message conditionally:
```typescript
} catch (err: unknown) {
  const reason = err instanceof Error ? err.message : String(err);
  diagnostics.push({
    code: "SPEC_SCHEMA_VERSION",
    message: `Migration from schema "${fromVersion}" to "1" failed: ${reason}`,
    severity: "warning",
    path: "" as JsonPointer,
  });
}
```
