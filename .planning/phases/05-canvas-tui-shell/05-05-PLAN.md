    `phase: PalettePhase` — starts as `{ kind: "filter" }`.

    On construction:
    - Build item list from `Object.values(COMMANDS)` sorted by noun prefix (D-76 grouping)
    - Noun prefix sort order: screen, component, entity, nav, action, then remaining alphabetically
    - Create `SelectList` with these items and SelectList `onSelect` wired to `startArgPrompt(commandName)`

    `handleInput(data: string)`:
    - Esc always → `this.onClose()` (no partial store.apply)
    - If `phase.kind === "filter"`:
      - Typing chars → delegate to `filterInput.handleInput(data)`, then `commandList.setFilter(filterInput.getValue())`
      - Enter → if item selected and has no required args: run immediately, close. Else: `startArgPrompt(commandName)`
      - Up/Down/j/k → delegate to `commandList.handleInput(data)` for navigation
    - If `phase.kind === "arg-prompt"`:
      - Enter → collect current arg value, advance to next arg or fire `store.apply` and close
      - Other keys → delegate to `this.argInput.handleInput(data)`

    `render(width: number)`:
    - Filter mode: `["> " + filterInput.render(width-2), ...commandList.render(width)]`
    - Arg-prompt mode: `["Arg " + phase.argKeys[phase.argIdx] + ": " + argInput.render(width - argLabelLen), ...]`
    - All lines: `truncateToWidth(line, width)` (Pitfall 1)
    - Max rendered height: ~15 lines (filter input + up to 10 command list items + arg prompts)

    `startArgPrompt(commandName: string)`:
    - Reads `COMMANDS[commandName].argsSchema.shape` to get arg key names
    - Filter to required args only (Zod shape keys where field is not `.optional()`)
    - If no required args: call `store.apply(commandName, {})` immediately and `onClose()`
    - Otherwise: set phase to `{ kind: "arg-prompt", commandName, argKeys: requiredKeys, argIdx: 0, collectedArgs: {} }`

    `invalidate()`: `filterInput.invalidate(); commandList.invalidate(); argInput.invalidate()`

    Do NOT implement `onSelect` on the SelectList for navigation — only for Enter selection. Use `commandList.handleInput(data)` to let the list manage its own j/k.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User-typed arg values → store.apply | Palette collects raw string from Input; passes to store.apply as args |
| COMMANDS enumeration | Palette reads from the Phase 4 COMMANDS registry — trusted internal data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-16 | Tampering | User-typed arg value flows to store.apply as string without pane-level validation | mitigate | store.apply validates via command.argsSchema.safeParse (T-04-01); palette collects and passes, store rejects invalid |
| T-05-17 | Tampering | Prototype pollution via arg key names from argsSchema.shape | mitigate | collectedArgs uses `Object.create(null)` pattern from T-04-23; key names come from Zod schema, not user input |
| T-05-18 | Denial of Service | Partial store.apply on Esc mid-arg-prompt | mitigate | Esc always calls onClose() without store.apply; no partial apply possible — store.apply called only when all required args collected |
| T-05-19 | Denial of Service | Stale palette instance reused after close → stale SelectList state | mitigate | Pitfall 5 documented: root.ts always creates `new CommandPalette()` on open; acceptance criteria verifies `new CommandPalette` in root.ts |
| T-05-20 | Tampering | Command name from SelectList item injected back to store.apply | accept | Command names come from COMMANDS registry keys (not user input); getSelectedItem().value is a known CommandName |
</threat_model>
